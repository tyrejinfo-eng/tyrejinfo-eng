# Contributing to TyRejinfo Engineering Projects

Thank you for your interest in contributing to the **TyRejinfo Engineering** ecosystem.

This organization includes projects such as:

* MonsterCoder
* MonsterTools
* AIBlue Copilot
* SGL-DevEngine

These projects are built around deterministic software engineering, modular architecture, and extensible tooling systems.

---

# 🚀 Code Philosophy

Before contributing, understand the core principle:

> **Deterministic systems execute software. Cognitive systems assist software.**

This means:

* AI does NOT directly control execution
* Workers perform deterministic operations
* Coordinators manage planning and execution flow
* Systems must remain predictable and reproducible

---

# 🧱 How to Contribute

## 1. Fork the Repository

Create your own fork of the repository you want to contribute to.

## 2. Create a Feature Branch

Use meaningful branch names:

```bash
feature/eventbus-improvements
fix/memory-leak-worker
docs/update-architecture-guide
```

---

## 3. Follow Code Standards

All contributions must follow:

* Clean architecture principles
* Deterministic behavior patterns
* Modular design
* SOLID principles where applicable
* Clear naming conventions
* No hidden side effects

---

## 4. Test Your Changes

Before submitting:

* Build the solution
* Run existing tests (if available)
* Verify no regressions
* Ensure deterministic behavior is preserved

---

## 5. Commit Guidelines

Use clear and meaningful commits:

```text
Add TaskEnvelope validation layer
Fix coordinator race condition in scheduler
Improve EventBus indexing performance
Refactor worker execution pipeline
```

Avoid vague commits like:

* "fix stuff"
* "updates"
* "changes"

---

## 6. Submit a Pull Request

When ready:

* Open a Pull Request
* Clearly describe your changes
* Include reasoning behind design decisions
* Link any related issues

---

# 🧠 Architecture Awareness

When contributing to MonsterCoder ecosystem projects, keep in mind:

* Orchestrator = execution authority
* Coordinators = cognitive planning layer
* Workers = deterministic execution units
* NuNu = guidance system only (never execution)
* Visual Runtime = structural representation, not decoration

Do not bypass system boundaries.

---

# 🚫 What NOT to do

* Do not introduce autonomous AI execution paths
* Do not bypass deterministic workers
* Do not mix UI logic with core runtime logic
* Do not introduce hidden side effects
* Do not break modular boundaries

---

# 🤝 Community Expectations

We expect contributors to:

* Be respectful

* Collaborate professionally
